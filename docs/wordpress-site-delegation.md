# WordPressサイト運用設計 — 他チームへの管理委譲

サイトの管理権限(コンテンツ編集)を基盤チーム以外のチーム(以下「サイトチーム」)へ
渡して運用する際の、管理方針の検討記録。個々の作業手順は各 `manual-*.md` を参照。

## 0. 大前提: この基盤では「構成」と「コンテンツ」で管理経路が違う

このリポジトリの設計上、**Gitで昇格(dev→staging→production)できるのは構成のみ**
(チャートバージョン、values、イメージdigest)。一方、記事・固定ページ・メディア・
プラグイン・テーマは **DBとwp-content(永続ボリューム)上のデータ**であり、
Gitの昇格フローには乗らない。環境間でデータを移すには
[manual-wordpress-restore.md](manual-wordpress-restore.md) の「DBとwp-contentを
丸ごと差し替える」手順しかなく、部分的な移送(記事1件だけ昇格、など)はできない。

したがって運用設計は次の4層に分けて考える。層ごとに担当・経路・環境が異なる。

| 層 | 具体例 | 実体 | 変更経路 |
|---|---|---|---|
| 基盤 | クラスタ、Longhorn、LB/IPPool、Secret | K8sリソース | Git(PR)+ 手動手順 |
| アプリ構成 | WPコア/PHPバージョン、レプリカ数、ボリュームサイズ | イメージdigest・チャートvalues | Git(PR)→ dev→staging→production 昇格 |
| サイト構成 | プラグイン、テーマ | **wp-content上のデータ**(ただし実質コード) | wp-admin / wp-cli(環境ごとに同じ操作を繰り返す) |
| コンテンツ | 記事、固定ページ、メディア、カテゴリ、WPユーザー | DB + wp-content/uploads | wp-admin(**本番に直接**) |

委譲するのは下2層(サイト構成・コンテンツ)のうち、原則**コンテンツ層のみ**。
サイト構成層(プラグイン・テーマ)は後述の申請フローで基盤チームが関与する。

## 1. 記事の追加はどこでさせるか → 本番に直接

**結論: 記事・メディアの追加/編集は本番のwp-adminで直接行わせる。**

- コンテンツはGit昇格の対象外。devで記事を書いても、本番へ持っていく手段は
  restore手順によるDB・wp-contentの**丸ごと上書き**しかなく、その間に本番側で
  行われた他の変更(コメント、別記事の編集、ユーザー追加)を破壊する。
  数十サイト規模でこの運用は成立しない。
- 「公開前レビュー」はWordPress自身の機能で賄う。環境を分けるのではなく
  **投稿ステータス(下書き→レビュー待ち→公開)とロール**で統制する:
  - 執筆者には `投稿者(Author)` または `寄稿者(Contributor)` を付与
    (寄稿者は自力で公開できず、レビュー待ちまで)。
  - サイトチームの責任者に `編集者(Editor)` を付与し、公開操作を担わせる。
- dev/stagingは**コンテンツ制作環境ではない**と位置づける。役割は
  「構成変更・プラグイン・バージョンアップの検証」。逆にデータの流れは
  本番→dev(restore手順で本番データの写しを検証用に流し込む)が正しい向き。

## 2. プラグイン・テーマの追加/更新 → 申請ベース、devで検査してから本番へ

**結論: サイトチームには直接操作させず、申請→基盤チームがdevで検査→本番適用。**

プラグインは実質コードであり、脆弱性・障害の主要な持ち込み口。かつこの構成では
プラグインファイルはwp-content(RWX永続ボリューム)上のデータなので、
**「devで入れてGitで昇格」はできない。昇格=各環境で同じインストール操作を
同じバージョンで繰り返すこと**を意味する。フローは以下。

1. **申請**: サイトチームがプラグイン名・バージョン・目的を申請(Issue等)。
2. **検査(dev)**: 基盤チームがdevの同サイトにインストールして確認:
   - 供給元・最終更新日・既知脆弱性(WPScan等)の確認。
   - **この構成特有の互換性**の確認(下記「検査観点」)。
3. **本番適用**: 検査で問題なければ、本番PVCのLonghornスナップショットを取った上で、
   基盤チーム(またはその場で権限を持つ者)が本番のwp-adminかwp-cliで
   **devと同一バージョン**をインストール/更新する。
4. **記録**: どのサイトに何のプラグインをどのバージョンで入れたかを台帳に記録する
   (wp-contentの中身はGitに残らないため、記録しないと構成が追跡不能になる)。
   台帳は本リポジトリ内(例: `docs/plugin-inventory.md` やサイトごとのREADME)でよい。

**更新も同じフロー**で行う。プラグインの自動更新は無効にし(放置すると本番が
検査なしで変わる)、月次などの定期メンテナンスウィンドウでまとめて
dev検証→本番適用する。セキュリティ緊急パッチのみ、スナップショット取得を条件に
本番先行を許容する。

### 検査観点(この構成特有のもの)

- **2レプリカ + RWX共有**: wp-contentは全レプリカ共有なのでファイルは即時一致するが、
  PHPセッションやローカルメモリに状態を持つプラグインはレプリカ間で不整合を起こす。
- **.htaccessは永続化されない**(`WORDPRESS_ENABLE_HTACCESS_PERSISTENCE: no`)。
  .htaccessに書き込むタイプのキャッシュ系・セキュリティ系・リダイレクト系プラグインは
  設定がApache再起動のたびに消える。
- **ファイルベースのキャッシュはNFS(Longhorn RWX)上で低速**。キャッシュ系は
  DB/オブジェクトキャッシュ方式を選ぶ。
- **コンテナは非rootかつコアディレクトリは書き込み不可**。wp-content外に
  ファイルを書こうとするプラグインは動かない。

### wp-adminからのWordPressコア更新は禁止

コア本体はコンテナイメージ(digest固定)に含まれ、永続化されるのはwp-contentのみ。
wp-admin上の「更新してください」通知からコアを更新しても、Pod再作成で消えるか
書き込み失敗になる。**コア・PHPの更新は基盤チームが
[images/wordpress/](../images/wordpress/) のdigest更新→チャート版数上げ→
dev→staging→production昇格で行う**([manual-multi-env.md](manual-multi-env.md) 4章)。
サイトチームには「管理画面の更新ボタンは押さない」ことをルールとして明示する。

## 3. 権限設計: 何を渡し、何を渡さないか

| 対象 | サイトチームに渡すもの | 渡さないもの |
|---|---|---|
| WordPress | 個人ごとのwp-adminアカウント(Editor以下) | `Administrator` ロール、初期 `admin` アカウントの共有 |
| インフラ | なし | kubectl / Rancher UI / Longhorn UI へのアクセス |
| GitHub | (必要なら)Issue起票のみ | 本リポジトリへのwrite権限 |

- **Administratorを渡さない**ことがプラグイン統制の実効性を担保する
  (Editor以下はプラグイン・テーマ・ユーザー・設定を操作できない)。
  これで「勝手にプラグインを入れない」がルールではなく権限で保証される。
- アカウントは**共有せず個人ごと**に発行する(監査ログ=誰がいつ何を変えたかの
  追跡と、退職・異動時の個別無効化のため)。発行・ロール変更・削除は
  Administrator権限を持つ基盤チームが行う。
- テーマファイル・プラグインファイルの管理画面内エディタは無効化しておく
  (wp-configに `DISALLOW_FILE_EDIT` を設定。新規サイトはチャートの
  `wordpressExtraConfigContent` 等で注入できるが、**wp-config.phpは初回生成後に
  再生成されない**ため、既存サイトはPod内で追記する。全サイト標準にするなら
  チャート側に入れてversionを上げ、通常の昇格フローで展開する)。

## 4. 委譲の前に整備が必要なもの(チェックリスト)

1. **SMTP設定**: 現状未設定のため、パスワードリセットも新規ユーザーへの招待メールも
   送信されない。**複数人にアカウントを配る運用の実質的な前提**。
   `helm.values.wordpress` の `smtpHost` 等で設定(認証情報はSecret化)。
2. **TLS/ドメイン**: 現状はLBのIPへhttp直アクセスのため、他チームのログイン認証情報が
   平文で流れる。委譲前に `ingress` + TLS(+DNS)への切り替えを強く推奨
   ([manual-wordpress.md](manual-wordpress.md) 補足参照)。
3. **バックアップ**: コンテンツは本番にしか存在しないため、バックアップが唯一の保険。
   Longhornの定期スナップショット/バックアップ(外部ターゲット)を本番の
   全サイトPVC(wp-content RWX + mariadb RWO)に設定し、RPO(何時間分の
   記事消失まで許容するか)をサイトチームと合意しておく。復旧は
   [manual-wordpress-restore.md](manual-wordpress-restore.md)。
4. **申請・連絡フロー**: プラグイン申請、アカウント発行依頼、障害連絡の窓口を決める
   (GitHub Issue推奨。テンプレートを用意するとよい)。
5. **サイトチーム向け利用ガイド**の配布。最低限:
   - 記事・メディアは本番のwp-adminで直接編集してよい。
   - 管理画面の「更新」通知(コア・プラグイン・テーマ)は操作しない。
   - プラグイン・テーマの追加/更新は申請する。
   - アカウントは共有しない。
   - 大規模な変更(テーマ切替、大量インポート)の前は基盤チームへ連絡
     (事前スナップショットを取るため)。
6. **プラグイン台帳**の初期化(現時点で入っているプラグインの棚卸し)。

## 5. 責任分担まとめ

| 作業 | 担当 | 経路 | 環境 |
|---|---|---|---|
| 記事・固定ページ・メディアの追加/編集 | サイトチーム | wp-admin(下書き→レビュー→公開) | 本番直接 |
| WPユーザーの発行・ロール変更 | 基盤チーム | wp-admin(Administrator) | 各環境 |
| プラグイン/テーマの追加・更新 | サイトチーム申請 → 基盤チーム実施 | wp-admin / wp-cli | dev検証 → 本番(同一バージョンを手動で反映) |
| WPコア・PHP更新 | 基盤チーム | Git(イメージdigest → チャート版数) | dev→staging→production 昇格 |
| サイト設定(レプリカ数、ドメイン、SMTP等) | 基盤チーム | Git(fleet.yaml / チャートvalues) | 昇格 |
| Secretローテーション | 基盤チーム | `seal-site-secrets.sh` + kubectl | 各環境 |
| バックアップ・リストア | 基盤チーム | Longhorn + 手順書 | 主に本番 |
| 障害一次対応 | 基盤チーム(サイトチームは通報) | kubectl / break-glass | 各環境 |

## 6. 見送った案(と理由)

- **「dev/stagingで記事を書いて本番へ昇格」**: コンテンツの部分昇格の仕組みが
  存在しない(restore手順は全上書き)。WordPress標準のレビューフローで代替できる。
- **サイトチームへAdministrator付与**: プラグイン統制がルール頼みになり、
  wp-content(Git外)の変更は事後検知も難しい。統制が権限で担保できなくなる。
- **プラグインをカスタムイメージやinitContainerに焼き込んでGit管理化**: 構成の
  Git化としては理想だが、Bitnamiの永続化構造(wp-contentがボリューム側)と相性が
  悪く、サイトごとのイメージビルドは数十サイト規模で運用負荷が大きい。
  将来プラグイン統制を強化したくなった場合の検討課題として残す。
